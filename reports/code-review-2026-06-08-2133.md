## 🔴 必须修改 (8项)

### 1. [models.js:2373行] 上帝组件 — 单文件超 2300 行，严重超限

`models.js` 将路由、状态管理、UI渲染、拖拽排序、API交互、模态框逻辑全部耦合在一个文件里。单文件 2373 行，远超 300 行合理阈值，难以维护和协作。

**建议**：按职责拆分为：
- `models-state.js` — 状态管理 + 配置读写 + 撤销栈
- `models-console.js` — 控制台渲染（主模型选择 + 一键方案）
- `models-fallback.js` — 备选瀑布流编辑器 + 拖拽排序
- `models-providers.js` — 服务商卡片列表 + CRUD
- `models-drag.js` — 共享拖拽排序逻辑（provider内 + fallback链共用）

### 2. [models.css:56-66 与 1397-1404] `.provider-card` 样式重复定义

`.provider-card` 及其子选择器在 CSS 文件中出现了**两次完整定义**（第 56-148 行 和第 1397-1483 行），属性值略有差异，产生覆盖冲突。同样 `.provider-card__header`、`.provider-card__chevron`、`.provider-card__name`、`.provider-card__body`、`.provider-card__batch`、`.provider-card__more` 也都是双重定义。

**建议**：删除第二套重复定义（1397-1483行），保留第一套并合并差异。

### 3. [models.css:470-488 与 1592-1599] `.btn-ghost` 样式重复定义

`.btn-ghost` 出现两次，属性略有不同（padding 值 0 8px vs 0 var(--space-2)），导致不确定最终应用的是哪个。

**建议**：统一为一个定义，使用 CSS 变量。

### 4. [models.js:65] `state` 对象无类型约束，内部属性分散

```js
const state = { config: null, search: '', undoStack: [], _fallbacks_expanded: false }
```

后续通过 `state._collapsed`、`state.sortBy`、`state._fallback_candidates_collapsed` 等动态挂载属性，没有任何初始化或类型约束。`_collapsed` 在 renderProviders 中首次访问时才 lazy init（第 934 行），`sortBy` 从未初始化默认值。

**建议**：
- 在 state 初始化时声明所有会用到的属性
- 添加 JSDoc 类型注释
- `sortBy` 默认值应显式设为 `'default'`

### 5. [models.js:448-497] `renderConsole` 全量 innerHTML 重建导致 Combobox 闪烁

每次状态变更后 `renderConsole` 用 `innerHTML` 整体重建 DOM，包括 Combobox 容器。虽然代码在重建后调用了 `_globalPrimaryCombo.destroy()` + 重新 `createModelCombobox()`，但这个「销毁→重建」周期会导致：
- 输入焦点丢失
- 下拉框闪烁
- 用户正在输入时被强制打断

**建议**：对 Console 区采用增量更新，只更新变更的文本节点和属性，而非整块 innerHTML。或至少对 Combobox 区域做 diff 更新。

### 6. [models.js:936-981] `renderProviders` 全量 innerHTML 导致交互状态丢失

每次搜索输入、折叠切换、排序变更都会触发 `renderProviders` 全量 innerHTML 替换，导致：
- 折叠/展开状态需从 `state._collapsed` 重建
- checkbox 勾选状态丢失（批量操作场景）
- 滚动位置重置
- Provider 操作菜单 hover 状态丢失

**建议**：至少将 innerHTML 全量替换改为针对性 DOM 更新；或将搜索过滤改为 CSS 隐藏而非移除 DOM。

### 7. [models.js:721-798 与 1203-1296] 拖拽排序逻辑完全重复

fallback 链拖拽（721-798行）和 provider 内模型拖拽（1203-1296行）是几乎完全相同的逻辑，仅选择器不同。这是典型的 DRY 违反。

**建议**：抽取 `createDragSortable(container, options)` 工具函数，两个场景共用。

### 8. [models.js:1017] 推理标签硬编码中文

```js
if (m.reasoning) badges.push(`<span class="model-tag model-tag--rz">推理</span>`)
```

这个 "推理" 没有走 i18n `t()` 函数，而同文件其他标签都用了 `t('models.reasoning')`。在非中文环境下会显示中文。

**建议**：改为 `t('models.reasoning')`。

## 🟡 建议优化 (10项)

### 1. [models.js:30-63] 页面结构信息层级不清

当前页面布局从上到下是：
1. 页面标题 + 顶部按钮
2. 晴辰云推广条（可折叠）
3. **控制台**（主模型选择 + 一键方案 + 备选pill）
4. **瀑布流编辑器**（备选链 + 候选池，可折叠）
5. **服务商列表**（搜索 + 卡片）

**体验问题**：
- 控制台是高频操作区（切模型），但被推广条推到了较下方
- 瀑布流编辑器与控制台底部的备选 pill 功能重叠，用户困惑「在哪管理备选？」
- 服务商列表是页面主体（占 80%+ 空间），但在最底部，新用户首屏看到的是控制台而非模型列表

**建议**：
- 将控制台做成页面顶部的固定/粘性面板（sticky）
- 合并控制台的备选 pill 和瀑布流编辑器入口，减少认知负担
- 考虑 Tab 分区：[模型路由] [服务商管理]，让两个核心功能区互不干扰

### 2. [models.js:438-497] 控制台区域信息密度过高

`.models-control-console` 在一个面板内同时承载了：
- 主模型图标 + 名称 + meta 信息
- Combobox 选择器
- 3 个操作按钮（测试/定位/应用到Gateway）
- 4 个一键方案按钮
- 推理模型开关
- 备选模型 pill 列表
- 管理备选按钮

用户需要在一屏内理解所有这些信息，认知负担较重。特别是「一键方案」和「推理开关」功能定位不同，但视觉权重相同。

**建议**：
- 将一键方案作为折叠/弹出面板，降低视觉噪音
- 推理开关放在 Combobox 旁边，与模型选择紧密关联
- 备选模型 pill 区域 + 管理按钮独立为一个子面板

### 3. [models.js:346-436] 瀑布流编辑器两栏布局在中等屏幕体验差

`.fallback-workbench` 使用 `grid-template-columns: minmax(0, 1fr) minmax(320px, 0.92fr)` 两栏布局。在 980px-1200px 之间，左栏（活跃链）被压缩到较窄，拖拽手柄和操作按钮空间不足。980px 以下变为单栏，但此时两面板上下排列过长。

**建议**：
- 考虑在中等屏幕用 Tab 切换「活跃链」/「候选池」
- 候选池默认折叠服务商组（仅当模型数 < 5 时展开），减少滚动

### 4. [models.css:538-541] 瀑布流两栏比例 1:0.92 不平衡

```css
grid-template-columns: minmax(0, 1fr) minmax(320px, 0.92fr);
```

左栏（活跃链，通常只有 2-3 项）和右栏（候选池，可能有数十项）视觉权重几乎相等。但活跃链内容少，候选池内容多，导致左栏大量空白。

**建议**：改为 `grid-template-columns: minmax(240px, 0.6fr) minmax(320px, 1fr)`，给候选池更多空间。

### 5. [models.js:959-967] Provider 操作菜单用 CSS hover 显示，触控设备不友好

```css
.provider-card__actions:hover .provider-actions-menu { display: flex; }
```

纯 hover 触发在移动端/触屏设备上无法使用。虽然代码中有 `:focus +` 备选，但用户需要知道去 focus 那个 toggle 按钮。

**建议**：改为 click 切换 + 点击外部关闭，同时兼容 hover 和 touch。

### 6. [models.js:924-925] 搜索和排序状态仅存在内存，切换页面即丢失

`state.search` 和 `state.sortBy` 不会持久化。用户在搜索框输入关键词后切换页面再回来，搜索状态丢失。

**建议**：将搜索和排序状态存入 URL query params（`?search=gpt&sort=latency-asc`），或至少用 sessionStorage。

### 7. [model-combobox.js:260] document 级 click 监听未在 destroy 中正确清理

```js
document.addEventListener('click', outsideClickHandler)
```

`destroy()` 中调用了 `document.removeEventListener('click', outsideClickHandler)`，但 `outsideClickHandler` 是闭包内的局部函数引用，这看起来是正确的。不过如果 `destroy` 不被调用（如页面切换时 cleanup 未触发），会泄漏。

**建议**：确保 `models.js` 的 `cleanup()` 函数在所有路由切换场景下都被调用；考虑用 `AbortController` 管理事件监听。

### 8. [models.js:1100-1144] `normalizeProviderUrls` 中的端点路径匹配过于激进

```js
for (const suffix of ['/api/chat', '/api/generate', '/api/tags', '/api', ...]) {
  if (url.endsWith(suffix)) { url = url.slice(0, -suffix.length); break }
}
```

用户如果确实需要访问 `/api` 结尾的自定义端点，会被静默截断。特别是 `/api` 作为 suffix 过于宽泛，可能误截合法 URL。

**建议**：
- 添加用户确认 toast
- 或只在首次检测到时自动修复，后续不再干预
- 考虑增加 `skipNormalization` 标记让高级用户绕过

### 9. [models.css:203-218] 瀑布流容器用 `max-height: 2000px` 做展开动画

```css
.models-waterfall-visible { max-height: 2000px; transition: max-height 400ms; }
```

`max-height` 动画的实际时长取决于内容高度与 2000px 的比例。如果内容只有 300px，动画会在 300/2000 * 400ms ≈ 60ms 内完成展开，其余时间都在"动画"不可见的空白区域。收起时更明显——先等 2000-300=1700px 的"空动画"，再真正收起。

**建议**：使用 JS 动态计算 `scrollHeight` 设为 `max-height`，或用 `grid-template-rows: 0fr → 1fr` 方案实现更自然的展开动画。

### 10. [models.css:886] `.connectors-filter` 用固定宽度 `flex: 0 0 180px`

在窄屏幕下，180px 固定宽度 + 搜索框的弹性宽度可能导致搜索框过窄。缺少对应的响应式断点处理。

**建议**：改为 `flex: 0 1 180px; min-width: 120px`，在窄屏幕时允许收缩。

## 🔵 可选优化项 (6项)

### 1. [models.js:456] Badge 文案混合中英

`fallbacks.length + ' ' + t('models.fallbackCount')` — 如果 `fallbackCount` 的中文翻译是"备选"，那结果是 "2 备选"。但 `totalModels` 翻译是"模型总数"，结果是 "5 模型总数"，语义不通。

**建议**：统一使用带占位符的 i18n key，如 `{count} 个备选` / `{count} 个模型`。

### 2. [models.js:86-137] `loadConfig` 中自动修复逻辑过于隐蔽

用户首次进入页面时，自动修复 URL + 自动切换主模型，只靠 toast 通知。如果修复导致意外行为，用户不知道如何回退。

**建议**：对破坏性自动修复（如主模型切换），改为显式确认对话框。

### 3. [models.css] CSS 变量混用两套命名体系

文件中同时使用：
- `--space-*` / `--radius-*` / `--font-size-*`（设计 Token 体系）
- `--text-1` / `--text-2` / `--text-3` / `--text-xs` / `--text-sm`（另一套体系）
- `--bg-hover` / `--bg-elevated` / `--bg-secondary`（又一套）
- `--border-1` / `--border-2` / `--border-primary` / `--border-secondary`（两套混用）

这导致同一语义的属性在不同地方用了不同的变量名，维护时容易混淆。

**建议**：统一为一套设计 Token 命名体系，逐步迁移。

### 4. [models.js:286-323] 一键方案算法隐式偏好

`getModelScore` 的加权公式是硬编码的（如 `latency / 2 - Math.min(context, 200000) / 120`），用户无法理解为什么某个模型被选为首选。

**建议**：在 UI 上展示评分依据（如 tooltip "延迟 1.2s · 上下文 128K"），或允许用户调整权重。

### 5. [models.js:40-44] 晴辰云推广条默认隐藏但入口不明显

`#btn-toggle-qtcool` 只是一个小 tag 按钮，新用户可能不知道点击它可以快速配置推荐服务商。

**建议**：首次进入时自动展开推广条（如果尚未配置任何服务商），或用更明显的 CTA 设计。

### 6. [model-presets.js:130-131] `fetchQtcoolModels` 硬编码 contextWindow

```js
return data.data.map(m => ({
  id: m.id, name: m.id, contextWindow: 128000,
  reasoning: m.id.includes('codex')
}))
```

所有远程获取的模型都设为 128K 上下文，推理模型判断仅靠 ID 包含 "codex"。这不够准确。

**建议**：尝试从模型 metadata 中读取实际 context window；推理模型判断可扩展关键词列表。

## 🟢 做得好的地方

1. **i18n 覆盖完整** — 除了一个硬编码"推理"外，所有 UI 文案都走了 `t()` 函数，多语言支持到位
2. **撤销机制** — `pushUndo` + `undoStack` 设计良好，20 步上限合理，给用户安全感
3. **自动保存防抖** — 300ms 防抖 + `noReload: true` 避免频繁写配置重启 Gateway
4. **Combobox 组件** — 键盘导航、拼音匹配、分组显示，交互体验专业
5. **错误处理** — `loadConfig` 的 catch 分支有完整的错误展示 + 技术详情折叠 + 重试按钮
6. **配置自动修复** — `normalizeProviderUrls` 处理了 Ollama 端口、尾部斜杠、错误 API 类型等常见坑
7. **拖拽排序** — Pointer Events 实现，兼容 Tauri WebView2/WKWebView
8. **CSS 响应式** — 有 640px / 980px / 1080px 三个断点，移动端适配较完善
9. **安全** — 所有动态内容都做了 `escapeHtml()`，无 XSS 风险

| 维度 | 评分 | 说明 |
|------|------|------|

## 风险等级：中

**总体评价：待修改后合并**

核心功能完整且安全，但布局和交互体验有几个需要优先解决的问题：
1. **上帝组件拆分**（影响长期可维护性）
2. **全量 innerHTML 重建**（影响用户操作流畅性）
3. **CSS 重复定义清理**（可能导致样式覆盖 bug）
4. **推理标签硬编码中文**（i18n 一致性）
